# k6 — ramping-arrival-rate for 7K EPS load test (confirmed)

## Scenario for statnive-live `/api/event` (7K EPS target)

```javascript
import http from 'k6/http';

export const options = {
  discardResponseBodies: true,   // critical for throughput

  scenarios: {
    ingest: {
      executor: 'ramping-arrival-rate',
      startRate: 1000,            // 1K iters/sec warm-up
      timeUnit: '1s',
      preAllocatedVUs: 200,       // pre-allocate — avoid mid-test VU spawn
      maxVUs: 500,                // hard cap
      stages: [
        { target: 1000, duration: '30s' },  // warm-up
        { target: 7000, duration: '2m'  },  // ramp to target
        { target: 7000, duration: '5m'  },  // sustain
        { target: 0,    duration: '30s' },  // ramp-down
      ],
      tags: { test_type: 'ingest' },
    },
  },

  thresholds: {
    'http_req_duration{test_type:ingest}': ['p(95)<200', 'p(99)<500'],
    'http_req_failed{test_type:ingest}':   ['rate<0.001'],   // <0.1% failure
  },
};

export default function () {
  http.post('https://staging.statnive.live/api/event',
    JSON.stringify({ /* event payload */ }),
    { headers: { 'Content-Type': 'text/plain' } }  // tracker sends text/plain to avoid preflight
  );
}
```

## Key findings from Context7

1. **`preAllocatedVUs` is NOT `maxVUs`.** docs explicitly warn: "You probably don't need maxVUs". Dynamic VU allocation during test causes generator overload. Set `preAllocatedVUs` generously.
2. **Scenario-specific thresholds** via tags: `'http_req_duration{test_type:ingest}'` — multiple scenarios can be asserted independently.
3. **`discardResponseBodies: true`** — mandatory for analytics tracker load tests. Response is 204, no body.
